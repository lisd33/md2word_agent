# 项目计划：结构-语义对齐的学术论文生成原型

## 1. 项目定位

本项目不是要做一个覆盖任意模板、任意版式、任意学科的“通用论文生成产品”，而是要完成一个**可验证的研究原型**，回答如下问题：

> 当学术论文模板的结构要求与用户内容意图不完全匹配时，系统能否进行合理的结构适配，并生成可继续编辑的文档草稿？

因此，整个计划围绕以下三点展开：

1. 提取模板的结构意图
2. 建模用户内容意图
3. 在两者之间进行结构-语义对齐

---

## 2. 范围控制

### 2.1 本阶段要做什么

- 支持一类受限期刊模板集合，例如计算机领域英文论文模板
- 从 `.docx` 模板中抽取核心结构信息
- 生成可编辑的中间表示与文档草稿
- 验证结构对齐是否优于 baseline

### 2.2 本阶段不做什么

- 不追求任意复杂 Word 版式的高保真还原
- 不追求自动生成高质量、可直接投稿的完整论文
- 不同时覆盖多种异构模板体系
- 不把 Web 产品化作为主要目标

### 2.3 推荐模板范围

为了控制 parser 难度和评测可比性，建议模板范围收敛为：

- **计算机领域英文论文模板集合**
- 优先考虑 `IEEE / ACM / Springer` 中的一个受限子集
- 不建议第一阶段同时支持所有出版体系

更稳妥的做法是先选择一个主模板族，例如：

- IEEE 类模板集合
- 或 ACM 类模板集合

只有在最小闭环稳定之后，再考虑跨模板族扩展。

---

## 3. 最小系统闭环

本项目的最小可运行流程不应只绑定在 `.docx` 上，而应抽象成“模板输入层 + 内容输入层 + 统一中间表示”的形式。

```text
输入层
  -> Word 模板 (.docx)
  -> 模板描述文件（JSON / YAML / Markdown）
  -> 自然语言模板规范 / 投稿要求
  -> 用户需求描述
  -> Markdown 提纲 / 草稿
  -> 纯文本摘要 / 研究想法

Input Router
  -> 识别输入类型
  -> 分发到 Template Parser / Rule Parser / Intent Parser
  -> 对多源输入做归并

统一表示层
  -> 模板结构需求表示
  -> 内容意图表示

模板结构需求 + 内容意图
  -> 结构-语义对齐决策
  -> 文档骨架生成
  -> 章节内容生成或补全
  -> 结构校验与轻量修正
  -> 可编辑输出（IR / HTML / Markdown / Word 简化导出）
```

这里需要明确两点：

1. `.docx` 模板解析仍然是当前阶段的重要输入来源，因为真实论文模板大多首先以 Word 模板出现。
2. `Markdown -> Word` 或 `Markdown -> 其他格式` 可以作为输出链路的一部分被支持，但**不是当前研究问题的核心贡献**。当前核心仍然是“模板结构约束与内容语义之间的对齐”。

换句话说，Markdown 在这个系统里应该被视为：

- 一种可接受的内容输入形式
- 一种可编辑中间表示或导出形式
- 但不是研究主线本身

其中真正的核心是：**结构-语义对齐决策**。

## 4. 技术路线

### 4.1 输入表示

系统输入不应假设只有单一 `.docx` 模板，而应支持异构输入并先做轻量路由。

- **Input Router**：负责识别输入属于模板文件、规则文本、内容草稿还是混合输入，并分发到对应模块
- **模板侧**：从 Word 模板或模板描述文件中提取章节层级、摘要/引言/方法/实验/结论等论文组成部分、图表槽位、引用风格等结构要求
- **规则侧**：从自然语言模板规范、投稿要求或排版说明中抽取显式结构与样式约束
- **内容侧**：从用户输入、Markdown 提纲或草稿中提取主题、写作类型、关键概念和是否需要实验/案例等意图信息

这里的 `Input Router` 是系统工程所需的前置模块，用于统一多种输入来源，但不作为本文的核心研究贡献。

### 4.2 中间表示

优先采用**结构化 JSON IR + 可渲染 HTML / Markdown**的组合方案：

- JSON IR 用于规划、校验和编辑
- HTML 用于可视化检查与简化导出
- Markdown 用于轻量编辑、提纲输入和部分输出场景

理由：

- JSON IR 比纯文本更适合程序操作和结构校验
- HTML 适合渲染检查和后续导出
- Markdown 适合承接用户已有提纲、草稿和轻量编辑需求
- 比直接操作 Word 更利于研究迭代

因此，这里的设计目标不是排斥 Markdown，而是避免把系统错误地等价为单纯的 `md2word` 工具。研究重点仍然是：在统一 IR 上进行结构约束建模和语义对齐。

### 4.3 模板理解与对齐策略

本项目采用“程序解析 + LLM 理解 + 规则约束”的混合方案，但三者的职责必须严格区分。

- **程序解析** 负责低层、确定性的工作，例如：读取 `.docx`、提取段落、样式、编号、图表位置、规则文本中的显式约束。
- **LLM 理解** 负责高层、语义性的工作，例如：判断一个候选标题到底是论文骨架章节、作者指南章节、排版说明章节还是元信息块；识别模板真正的结构意图；将候选结构归一化为可用的模板骨架。
- **规则约束** 负责系统边界和一致性检查，例如：必备章节不能丢、层级不能乱跳、引用风格必须继承、输出 schema 必须合法。

也就是说，LLM 不替代底层 parser，但必须参与模板结构意图理解；否则系统只能“检测标题”，无法“理解模板”。

## 5. 阶段化推进方案

项目不按固定周次推进，而按**阶段目标是否完成**来推进。只有当前阶段的关键产出达到要求，才进入下一阶段。

### 阶段 0：问题收敛与边界冻结

**阶段目标**

把项目从“想做很多功能”收敛成“验证一个核心研究问题的原型”。

**要解决的问题**

- 到底支持哪一类模板
- 到底研究什么，不研究什么
- 最终要证明的效果是什么

**核心任务**

- 确定唯一主场景，例如受限的计算机领域英文期刊/会议模板集合
- 冻结研究问题表述、方法边界和系统范围
- 明确输入类型集合，例如：模板文件、规则文本、内容草稿、混合输入
- 确定核心结构元素，只保留：章节层级、摘要与参考文献等论文组成部分、图表槽位、引用风格
- 写清楚 baseline 和评测目标，避免后面边做边改题目

**阶段产出**

- `RESEARCH_PROBLEM.md`
- `PROJECT_PLAN.md`
- `docs/dataset_scope.md`
- `docs/evaluation_scope.md`

**退出条件**

- 模板范围已冻结
- 不再继续扩展研究问题
- 已有一版可执行的最小系统定义

### 阶段 1：模板数据集与标注规范

**阶段目标**

建立一个足够支持研究验证的小规模模板集，并形成可复用的结构标注规范。

**输入**

- 冻结后的场景定义
- 待收集的模板来源范围

**核心任务**

- 收集 8-10 个同类模板，优先保证结构风格相近但实现方式不同
- 记录模板来源、用途、出版体系、格式版本
- 如有配套作者指南或模板说明，同步收集规则文本
- 人工分析每个模板的结构特征
- 制定结构标注规范，明确：
  - 什么算结构元素
  - 什么算装饰元素
  - 章节层级如何判断
  - 图表槽位如何标记
  - 摘要、参考文献等组成部分如何识别
- 为模板编写标注文件，至少覆盖核心字段
- 整理 2-3 组典型内容场景，用于后续对齐实验

**建议数据组织**

- `data/templates/raw/`
- `data/templates/README.md`
- `data/annotations/`
- `docs/annotation_guideline.md`
- `docs/template_taxonomy.md`

**阶段产出**

- 一套受限期刊模板集合
- 一份结构标注规范
- 一批模板结构标注结果
- 一份模板结构类型总结

**退出条件**

- 模板集足够支持后续 parser 开发和评测
- 标注规范能稳定指导第三方或未来自己重复标注
- 至少已经明确几种典型结构差异和冲突场景

### 阶段 2：候选结构抽取与中间表示

**阶段目标**

把多源模板输入转换成“候选结构集合”和“显式约束集合”，为后续的大模型结构意图理解提供可靠输入。

**输入**

- 模板数据集
- 规则文本样本
- 结构标注规范

**核心任务**

- 设计最小 IR，覆盖：
  - 文档元信息
  - 摘要与参考文献等组成部分
  - 章节候选树
  - 图表/表格候选槽位
  - 引用风格候选
- 定义 IR 的序列化格式和 JSON Schema
- 实现 Input Router，将输入分发到 Template Parser / Rule Parser / Intent Parser
- 实现基础 Word 读取器
- 实现段落样式、编号和局部上下文提取
- 实现标题候选检测器：
  - 基于样式名
  - 基于编号模式
  - 基于层级推断
- 实现组成部分候选提取与装饰元素初筛
- 实现图表槽位和表格位置的基础提取
- 实现 Rule Parser，从规则文本中抽取显式约束
- 记录候选结构及其局部上下文，而不是过早把它们当成最终模板骨架
- 做 parser / router 单元测试和误差分析

**建议代码结构**

- `src/ir/models.py`
- `src/ir/schema.json`
- `src/input/router.py`
- `src/parser/base_reader.py`
- `src/parser/heading_parser.py`
- `src/parser/figure_parser.py`
- `src/parser/rule_parser.py`
- `src/parser/to_ir.py`
- `tests/parser/`
- `tests/input/`

**阶段产出**

- 一套最小 IR 定义
- 一个可运行的模板候选结构解析器与输入路由器
- 文件侧和规则侧的初步约束提取流程
- 一组 parser / router 测试用例和错误案例

**退出条件**

- 大多数模板能被成功解析为候选结构 IR
- 常见规则文本能被稳定抽取为显式约束
- 已知错误模式被记录，而不是隐性存在
- 候选结构及其上下文可以直接喂给后续的 LLM 结构理解模块

### 阶段 3：模板结构意图理解与模板侧规范构建

**阶段目标**

让系统不只是“抽出标题”，而是真正理解模板的结构意图，区分哪些候选结构属于论文骨架，哪些只是作者指南、排版说明或元信息。

**输入**

- 阶段 2 产生的候选结构 IR
- 规则文本中的显式约束
- 模板局部上下文样本

**核心任务**

- 定义模板候选结构的语义角色标签，例如：
  - `structural_section`
  - `instructional_section`
  - `formatting_guideline`
  - `metadata_block`
  - `reference_block`
- 设计 LLM 输入表示，使模型能同时看到：
  - 候选标题文本
  - 标题层级
  - 样式/编号线索
  - 邻近段落内容
  - 来自规则文本的显式约束
- 实现基于 LLM 的模板结构意图分类器 / skeleton filter
- 让模型输出：
  - `keep / drop / normalize`
  - canonical section label
  - 简短解释
- 将过滤后的结构归一化成模板骨架，而不是保留所有 heading-like 候选
- 将文件侧结构理解结果与规则文本中的显式约束合并，形成统一的 `TemplateRequirement`
- 整理典型错误场景：
  - 作者指南章节被误当成论文骨架
  - 排版说明章节被误保留
  - 同一章节的不同写法未被归一
  - 文件侧和规则侧约束冲突

**这一阶段必须引入 LLM**

原因不是为了替代 parser，而是因为“模板结构意图理解”本身就是语义任务。像 `III. MATH` 这类标题是否属于真实论文结构，不能只靠硬编码判断。

**建议代码结构**

- `src/planner/template_understanding.py`
- `src/planner/template_filter.py`
- `src/planner/requirement.py`
- `src/planner/prompting.py`
- `docs/template_intent_cases.md`

**阶段产出**

- 模板候选结构的语义角色定义
- 一个基于 LLM 的模板 skeleton filter / section classifier
- 统一的模板侧 `TemplateRequirement`
- 一批模板理解错误案例与修正记录

**退出条件**

- 系统能将候选标题稳定过滤成可用的论文骨架
- instructional / structural 的区分达到基本可用水平
- 输出的 `TemplateRequirement` 可以被后续内容侧模块直接使用

### 阶段 4：内容意图建模与结构-语义对齐

**阶段目标**

在已经理解模板结构意图的前提下，完成项目最核心的方法模块：给定模板侧 `TemplateRequirement` 和用户内容意图，输出可解释的结构适配结果。

**输入**

- 阶段 3 产生的统一模板侧 `TemplateRequirement`
- 用户输入样例和内容场景

**核心任务**

- 定义内容意图表示：
  - 主题领域
  - 写作类型
  - 关键概念
  - 是否包含实验/案例/图表
  - 内容侧约束
- 实现内容意图抽取模块
- 设计结构对齐动作空间，例如：
  - `keep`
  - `optionalize`
  - `remove`
  - `add`
  - `split`
  - `merge`
- 实现结构-语义对齐模块
- 为对齐结果生成简短理由，保证可解释性
- 整理典型冲突场景：
  - 模板要求实验，内容偏理论
  - 模板要求多个图表，内容图表不足
  - 模板章节粒度过粗或过细
  - 用户内容需要新增中间章节
- 设计 baseline planner，用于后续对比

**如果要引入 7B 小模型，这一阶段仍然是主战场之一**

小模型适合做：

- 内容意图分类
- 章节动作预测
- 对齐后骨架生成
- 受约束 JSON 输出

但这里的前提是：模板侧结构已经经过阶段 3 的 LLM 结构理解，而不是未经筛选的 raw headings。

**建议代码结构**

- `src/planner/intent.py`
- `src/planner/alignment.py`
- `src/planner/prompting.py`
- `docs/alignment_cases.md`

**阶段产出**

- 内容意图表示方案
- 结构对齐算法或模型
- 对齐动作定义与推理流程
- 一批 case study 和失败案例

**退出条件**

- 系统能在典型冲突场景下输出合理的结构调整方案
- 输出结果有一致的格式和可解释理由
- 相比简单模板填充，已经表现出明显优势

### 阶段 6：评测与基线对比

**阶段目标**

用定量和定性证据证明：结构-语义对齐方法相对 baseline 是有效的。

**输入**

- 冻结版本的系统
- 模板集与测试场景
- baseline 实现

**核心任务**

- 定义自动评测指标：
  - 章节层级准确率
  - 必备章节召回率
  - 多余章节误报率
  - 图表槽位匹配率
  - 引用编号一致性
  - 标题-内容相关性
- 实现并运行 baseline
- 统计结果并做显著失败案例分析
- 设计人工评测表单
- 组织小规模用户研究，重点比较：
  - 手工搭骨架
  - 系统生成骨架后人工修改
- 收集时间数据与主观反馈

**正式 baseline 设计**

1. **Baseline 1: Direct Generation**
   给定用户输入和模板文字说明，直接让大模型生成论文结构或草稿，不显式执行模板解析与结构对齐。

2. **Baseline 2: Template Filling**
   将模板视为固定槽位集合，按照预定义规则填充内容，不允许根据语义动态增删章节。

3. **Baseline 3: Ours w/o Alignment**
   使用相同的模板解析和文档生成流程，但移除结构-语义对齐模块，直接从模板结构进入后续生成。

4. **Ours**
   完整使用模板解析、内容意图建模、结构-语义对齐、文档生成与校验流程。

**这些 baseline 分别回答的问题**

- `Direct Generation`：证明显式结构建模是否必要
- `Template Filling`：证明动态适配是否优于机械模板填充
- `Ours w/o Alignment`：证明对齐模块本身是否带来增益

**建议目录结构**

- `evaluation/metrics.py`

### 阶段 7：论文写作与研究包装

**阶段目标**

把系统实现沉淀成清晰的论文叙事，而不是简单开发记录。

**输入**

- 系统设计
- 实验结果
- 失败案例与边界分析

**核心任务**

- 写清楚问题定义、动机和研究边界
- 整理相关工作，明确与文本生成、模板填充、格式转换、Design-as-Code 的区别
- 抽象方法框架和系统架构图
- 撰写实验部分，突出：
  - 方法有效性
  - 适用边界
  - 失败模式
- 整理案例图、表格和附录
- 完成论文初稿与反复修订

**建议目录结构**

- `paper/sections/intro.tex`
- `paper/sections/related.tex`
- `paper/sections/method.tex`
- `paper/sections/experiments.tex`
- `paper/sections/discussion.tex`
- `paper/main.pdf`

**阶段产出**

- 论文初稿
- 图表与案例材料
- 最终答辩或汇报材料

**退出条件**

- 论文叙事能清楚回答研究问题
- 方法、实验和结论三部分逻辑闭环

## 6. 评测设计

### 6.1 自动评测

自动评测优先回答两个问题：

1. 生成结构是否更接近模板要求
2. 生成内容是否更符合章节语义和用户意图

建议指标：

- 章节层级准确率
- 必备章节召回率
- 多余章节误报率
- 图表槽位匹配率
- 引用编号一致性
- 标题-内容相关性得分

### 6.2 人工评测

建议人工评分维度：

- 结构合理性
- 与用户需求的匹配程度
- 后续编辑便利性
- 作为初稿的可接受程度

### 6.3 失败案例分析

必须专门整理以下失败类型：

- 模板结构抽取错误
- 内容意图识别错误
- 对齐决策过度保守
- 对齐决策过度激进
- 生成后结构漂移

## 7. 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 模板体系差异过大 | Parser 不稳定 | 限制模板集合，只支持一个受限模板族 |
| 对齐逻辑不够稳 | 研究核心失效 | 先做典型 case，再逐步扩展 |
| 导出质量不稳定 | 难以展示系统结果 | 将 Word 导出降级为简化目标 |
| LLM 内容质量波动 | 实验噪声大 | 聚焦结构评测，弱化内容文学性比较 |
| 范围膨胀 | 研究失控 | 冻结非核心功能，如 Web UI 和复杂反射 |

---

## 8. 当前最优先的动作

1. 冻结模板范围，例如 IEEE / ACM / Springer 中的一个子集
2. 建立模板集和标注规范
3. 先打通一个模板的 `Word -> IR -> HTML` 最小闭环
4. 尽快进入结构-语义对齐模块的验证

如果阶段 2 之前还没有跑通最小闭环，就不要过早讨论大规模评测或产品化界面。
